# PR 1360: DSL Preprocessing

This PR introduces a C++-based preprocessing layer (`src/dsl-preprocess.cpp`)
that transforms NONMEM-style model code into valid C++ that can be used in 
mrgsolve. This does not change the users source code; rather, modifications 
are made under the hood and only appear in the `.cpp` file that is actually
compiled; these changes are on top of a host of changes that mrgsolve is already 
doing. The difference is that the current work is using compiled Boost 
functionality rather thand regex.

## Exponents: `**` to  `pow()` conversion

The first piece was converting Fortran-style exponentiation (`**`) to C++
`pow()` calls. The implementation uses a Boost.Spirit parser to build a full
expression AST rather than doing regex substitution, so operator precedence and
parenthesization are handled correctly. Initial prototyping used a series of R
functions to do this; it didn't work that bad, but I could see that there would
be edge cases that would be difficult to handle this this approach.

A warning is issued if a line contains `**` but the parser cannot convert it
(e.g., due to an unrecognized syntax), so the user is told rather than silently
left with bad code. The original line is passed through unchanged when
conversion fails.

This runs by default for `$MAIN`, `$TABLE`, `$ODE`, `$EVENT`, `$PREAMBLE`, and
`$PRED` blocks (see `GLOBALS$PRE_PROC_BLOCKS`). It can be turned off via an
environment variable (`MRGSOLVE_CONVERT_POW = FALSE`) in `$ENV`.

While it is possible to have `**` in C++ code, it should be very rare; so I feel
good about processing this by default. If the user never has `**` in their code,
this will do nothing. 

There is an exported function called `convert_pow()` that will process a
character vector of code.

## Integer-division warning

Building on the same AST infrastructure, the preprocessor walks parsed
expressions looking for division of two bare integer literals (e.g. `3/4`,
`1/2`). In C++, these truncate toward zero, so `3/4` evaluates to `0`. This is a
big problem for people coming from NONMEN world, which handles `3/4` as `0.75`;
we've had senveral complaints of users getting bit by this, despite warnings in
the user guide. A warning is issued for each occurrence, with a suggestion to
use `3.0/4.0` instead. 

This also runs by default across the same blocks and can be disabled with
`MRGSOLVE_WARN_INT_DIV = FALSE` in `$ENV`.

I think it's the right decision to warn by default on this; can't imagine where
you would really need integer `3/4` and there are ways to accomplish this in the
code or just set the escape env var to disable.

There is an exported function called `warn_int_div()` that will process a
character vector of code. 

## Fortran IF/THEN/ELSE/ENDIF conversion

Fortran block-structured conditionals and single-line `IF` forms are converted
to C++. For example:

```
IF (WT .GE. 90) THEN
  CL = CL * 0.8
ENDIF
```

becomes:

```cpp
if(WT >= 90) {
  CL = CL * 0.8;
}
```

This handles `ELSEIF`/`ELSE IF`, single-line `IF(cond) statement` forms, bare
`ELSE` and `ENDIF`/`END IF`, and converts all Fortran relational and logical
operators (`.GE.`, `.LE.`, `.GT.`, `.LT.`, `.EQ.`, `.NE.`, `.AND.`, `.OR.`,
`.NOT.`, `.TRUE.`, `.FALSE.`) case- insensitively. The conversion runs only when
the `nm-vars` plugin is active (see below), and can be disabled via
`MRGSOLVE_CONVERT_FORT_IF = FALSE` in `$ENV`.

Like the `convert_pow()` functionality, if the user never uses these constructs,
the processor should just pass the code through. This also will never touch code
that doesn't invoke `nm-vars` plugin. The environment variable escape route is
provided just in case we have some false positive detection of code to modify.

There is an exported function called `convert_fort_if()` that will process a
character vector of code.

## Semicolon insertion

A semicolon inserter adds trailing semicolons to statement lines that lack them,
to support NONMEM-style code where semicolons are optional. Lines ending with a
continuation operator (`+`, `-`, `*`, `/`, `&`, `,`, `=`, `(`) are left alone —
this covers multi-line expressions. Lines that are blank, brace-only,
preprocessor directives, or C/C++ comments are also left untouched.

An important edge case: Fortran uses `;` as an inline comment delimiter. If a
`;` is found mid-line and nothing after it looks like C++ (no further `=` or
`;`), the text after the semicolon is stripped as a Fortran comment. This
preserves compatibility when someone has Fortran-style inline comments in their
code.

Semicolon insertion is off by default. It is enabled by adding the `semicolons`
plugin (or the composite `nm-like` plugin) to `$PLUGIN`. It only operates in
combination with `nm-vars`.

This will be very helpful if it works right. And it should be fine as long as
someone is truly coming from NONMEM. This is why you not only have to be using
`nm-vars` but you also need to opt in through the `semicolons` plugin.

There is an exported function `convert_semicolons()` that will process a
character vector of code.

## Plugin integration: `nm-like` composite plugin

A new `nm-like` composite plugin was added as a convenience: invoking `$PLUGIN
nm-like` automatically pulls in `nm-vars`, `autodec`, and `semicolons`. This
makes it easier to write fully NONMEM-style model code without individually
listing the component plugins. 

I think making the `semicolons` available along with the `IF/ELSE` processing, 
we need to emphasize that, when you're in NONMEM mode, you really need to 
commit to those patterns. Veering off into more complex C++ stuff could 
possibly not end well. I'm fine with this; IMO, invoking `nm-vars` is the 
user opting into this restriction; this can't be done lightly.

## User-facing R functions

The four preprocessing functions are exported and documented together under
`?dsl_preprocess`:

- `convert_pow(x, block = "")` — convert `**` to `pow()`
- `warn_int_div(x, block = "")` — warn about integer division
- `convert_fort_if(x)` — convert Fortran IF constructs
- `convert_semicolons(x)` — insert missing semicolons

These can be called directly on character vectors for testing or interactive
inspection.

## Environment variable control

All three default-on preprocessors can be turned off from within a model file
using `$ENV`:

```r
$ENV
MRGSOLVE_CONVERT_POW      = FALSE
MRGSOLVE_CONVERT_FORT_IF  = FALSE
MRGSOLVE_WARN_INT_DIV     = FALSE
```

The flags are read from `$ENV` during `parse_env()` when `mread()` processes the
model, so they apply on a per-model basis rather than globally.
